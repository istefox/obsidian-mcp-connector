# ADR-0013: Pure-Node self-contained .mcpb shim (no npx/mcp-remote)

**Status:** Proposed
**Date:** 2026-07-15
**Deciders:** Stefano Ferri

---

## Context

The generated `.mcpb` extension (`mcpbGenerator.ts`, `buildShim()`) ships a
`server/index.js` shim that, after resolving the plugin's port and bearer
token from `data.json`, does:

```js
spawn("npx", ["-y", "mcp-remote", url, "--header", `Authorization: Bearer ${token}`], { stdio: "inherit" })
```

Claude Desktop launches this shim with its own bundled Node runtime, but the
child process it spawns inherits Claude Desktop's GUI-launched environment,
whose `PATH` is the OS default (`/usr/bin:/bin:/usr/sbin:/sbin` on macOS),
with no Homebrew, no nvm, no global npm bin dir. `npx` is not on that `PATH`.
`spawn` fires an unhandled `error` event (`ENOENT`), the shim process exits
immediately, and Claude Desktop reports "Server transport closed
unexpectedly" with no useful stderr. Reproduced deterministically:

```
env PATH=/usr/bin:/bin:/usr/sbin:/sbin node server/index.js   # crash
env PATH=$PATH node server/index.js                            # works
```

This is not fixable by pre-warming the npx cache (`preWarm.ts`) or detecting
Node (`nodeDetect.ts`). Those address a slow first `npx` download, not a
missing `npx` binary on the child's `PATH`. The only durable fix is to stop
spawning a child process at all: the shim becomes a self-contained stdio ⇄
HTTP proxy using only Node built-ins available to whatever `node` binary
Claude Desktop already used to launch it.

The Obsidian-side HTTP server (`mcp-transport`) is stateless
(`sessionIdGenerator: undefined`), POST-only, and, since 0.22.0, may
answer a POST with `text/event-stream` instead of `application/json` when
the request is `activate_tool`/`activate_tools` (so a
`tools/list_changed` notification can ride along with the result). Any
stdio bridge that assumes `application/json` breaks on those calls; this is
exactly the bug `scripts/obsidian_mcp_bridge.py` fixed for Windows in
ADR-0012. The new Node shim needs the same SSE-parsing capability the
Python bridge already has, translated to JavaScript.

### Hard constraints (from SPEC.md, non-negotiable)

- Node stdlib only in the shipped shim (`node:fs`, `node:net`, global
  `fetch`). No dependencies, no build step for the shim's own execution.
  It runs verbatim under Claude Desktop's bundled Node (≥18; Claude Desktop
  ships v22).
- No TypeScript syntax in the shipped shim. It is executed directly by
  `node`, not transpiled at spawn time.
- `.mcpb` manifest structure (`mcp_config: node ${__dirname}/server/index.js`,
  fixed in 0.26.2) is unchanged. Only `server/index.js`'s contents change.
- Per-request `data.json` resolution (port/token), 30 s connect-retry
  window, 60 s per-request timeout, JSON-RPC `-32000` errors with
  stay-alive semantics, stderr lifecycle/error logging
  (`obsidian-mcp-connector:` prefix, `OBSIDIAN_MCP_DEBUG=1` for verbose,
  never log payloads).

## Decision

### 1. Shim source lives as a real, standalone `.js` file; a checked-in generated TS module carries it into the runtime bundle

`connectorShim.js`
(`packages/obsidian-plugin/src/features/mcp-client-config/services/`) is a
plain CommonJS Node script, the literal bytes Claude Desktop executes,
modulo one placeholder substitution (see §3). It is directly importable and
directly executable:

- `bun test` imports it and calls its pure functions with no mocking of
  sockets or the filesystem for the parsing/framing/error-building logic.
- `node connectorShim.js` (with a real `data.json` on disk) runs it as the
  actual shim, for manual smoke testing outside the `.mcpb` zip.

The problem this has to solve is distribution: `mcpbGenerator.ts` runs
**inside the already-bundled `main.js`**, at Obsidian runtime, when the user
clicks "Download .mcpb". `main.js` is a single file produced by
`bun.config.ts` (`Bun.build`, `entrypoints: ["./src/main.ts"]`). A file that
is not reached via that import graph does not exist inside Obsidian's
plugin folder at runtime. `fs.readFileSync("./connectorShim.js")` from
inside `main.js` would fail; the loose `.js` file is a repo artifact, not a
shipped one.

This project already solved the identical problem for the `.mcpb` icon:
`assets/iconPng.ts` is a checked-in, auto-generated TS module exporting the
icon's bytes as a base64 string constant (`ICON_PNG_B64`), regenerated by a
documented one-liner, imported normally by `mcpbGenerator.ts` so `Bun.build`
inlines it into `main.js` like any other string literal. `connectorShimSource.ts`
(same `assets/` folder) follows the same shape:

```ts
// AUTO-GENERATED — do not edit by hand.
// Source: packages/obsidian-plugin/src/features/mcp-client-config/services/connectorShim.js
// Regenerate: bun run packages/obsidian-plugin/scripts/gen-shim-source.ts
export const CONNECTOR_SHIM_SOURCE = "#!/usr/bin/env node\nconst fs = require(\"fs\");...";
```

`scripts/gen-shim-source.ts` reads `connectorShim.js`, `JSON.stringify`s the
full text (safe against the backticks and `${...}` template literals the
shim's own source contains; a raw template-literal embed is not, see
Alternatives), and writes the module above. A test in `mcpbGenerator.test.ts`
re-reads `connectorShim.js` from disk at test time and asserts it equals
`CONNECTOR_SHIM_SOURCE` byte-for-byte: a stale-generation guard that fails
loud (with a "run the regen script" message) instead of silently shipping a
`.mcpb` whose behavior no longer matches the tested, reviewed source. This
guard is the one thing the icon precedent doesn't need (icons don't get
edited casually; shim logic will, across future bugfixes) and is the reason
this ADR calls out the codegen step explicitly rather than treating it as
implied by the icon precedent alone.

`mcpbGenerator.ts` imports `CONNECTOR_SHIM_SOURCE` and no longer contains
shim logic itself. `buildShim()` shrinks to a single placeholder
substitution (§3).

### 2. Layered design inside `connectorShim.js`: pure core, thin impure shell, injectable I/O seams

Mirrors ADR-0012's pipeline (`parse_sse` / `route_sse_messages` /
`resolve_response_messages`, pure; I/O confined to a thin `main()`),
translated to JS and extended with explicit dependency-injection seams so
the impure shell is unit-testable too, matching how the Python bridge's own
tests mock `urllib.request.urlopen` and inject `stdin`/`argv` into `main()`:

**Pure (no I/O, exported for direct testing):**

- `splitLines(remainder, chunk) -> { lines, remainder }`: stdin chunk
  reducer; complete newline-terminated lines out, partial tail carried to
  the next call. Decouples line-framing from the actual `'data'` event
  handler.
- `parseJsonRpcLine(line) -> { message } | { error }`: parse + shape-check
  one line. Unparseable line -> `{ error }`, logged to stderr, dropped (no
  `id` to answer against, per SPEC edge cases).
- `parseTransportFile(jsonText) -> { port, token } | { error }`: validates
  `mcpTransport.livePort`/`mcpTransport.bearerToken` are present; does not
  touch the filesystem.
- `parseSse(body) -> message[]`: direct JS port of the Python bridge's
  `parse_sse`: buffers `data:` lines, dispatches on blank line or end of
  body, JSON-decodes each joined block, drops (does not throw on) an event
  whose data isn't valid JSON. Same three line-ending forms (`\r\n`, `\r`,
  `\n`).
- `routeSseMessages(messages, requestId) -> { response, notifications }`:
  splits an SSE body's messages into the one matching `requestId` (if any)
  and everything else, forwarded as notifications.
- `resolveResponseMessages(contentType, rawBody, requestId, status) -> message[]`:
  the single entry point the impure fetch layer calls: dispatches on
  `content-type` (`application/json` vs `text/event-stream`), or builds a
  `-32000` error message for a non-2xx status, an empty body, or an
  unparseable body. Same signature shape as the Python bridge's function of
  the same name (adapted to JS naming), for direct side-by-side review.
- `buildErrorResponse(id, message, data) -> object`: the `-32000` JSON-RPC
  error envelope, `obsidian-mcp-connector:`-prefixed message.

**Impure (I/O, each taking its real dependency as a parameter with a
production default, the same seam Python's tests use via `mock.patch` and
an injectable `stdin=`):**

- `readTransport(dataPath, { readFileSync = fs.readFileSync } = {})`: reads
  and delegates to `parseTransportFile`.
- `probePort(port, { createConnection = net.createConnection } = {})`: one
  raw TCP connect attempt, 1 s timeout.
- `resolveTransportWithRetry(dataPath, deadlineMs, { ...same seams, sleep } = {})`:
  the folded-in `waitForServer` logic: read + probe; on failure, sleep 1 s,
  retry, bounded by `deadlineMs` (30 000). Called reactively (see §4), not
  at startup.
- `postJsonRpc(url, token, message, timeoutMs, { fetchImpl = fetch } = {})`:
  one POST, `AbortController`-based timeout, returns `{ status, contentType, body }`
  or throws a typed connection-error.
- `runMain({ stdin = process.stdin, writeChunk = (s) => process.stdout.write(s), fetchImpl = fetch, dataPath, now = Date.now, sleep, log = console.error } = {})`:
  the event loop: frames stdin via `splitLines`/`parseJsonRpcLine`, dispatches
  each message (request vs. notification, by presence of `id`), on stdin
  `'end'` awaits all in-flight promises (each already bounded by its own
  timeout) then resolves so the caller can `process.exit(0)`.

`main()` at the bottom of the file calls `runMain()` with all-production
defaults, gated by `if (require.main === module) { main(); }`. Importing
the file for tests never starts the real stdin loop, mirroring the Python
bridge's `if __name__ == "__main__":` gate (an existing project convention,
per prior-agent notes on ADR-0012).

### 3. Vault-path parameterization: a single placeholder token, substituted and guarded

```js
// in connectorShim.js
const vaultPath = "__OBSIDIAN_MCP_VAULT_PATH__";
```

```ts
// in mcpbGenerator.ts
const VAULT_PATH_PLACEHOLDER = '"__OBSIDIAN_MCP_VAULT_PATH__"';

function buildShim(input: McpbGeneratorInput): string {
  if (!CONNECTOR_SHIM_SOURCE.includes(VAULT_PATH_PLACEHOLDER)) {
    throw new Error(
      "connectorShim.js is missing the vault-path placeholder — regenerate assets/connectorShimSource.ts",
    );
  }
  return CONNECTOR_SHIM_SOURCE.replace(
    VAULT_PATH_PLACEHOLDER,
    JSON.stringify(input.vaultPath),
  );
}
```

Same mechanism as today (`JSON.stringify(input.vaultPath)` injected as a
JS string literal, which handles embedded quotes/backslashes in Windows paths
correctly), just operating on a loaded string instead of a template
literal. The presence check turns a silently-wrong future edit (someone
renames the placeholder inside `connectorShim.js` without updating
`mcpbGenerator.ts`) into a build/test-time throw instead of a `.mcpb` that
ships with a literal, non-functional `__OBSIDIAN_MCP_VAULT_PATH__` string
baked in.

The plugin folder id (`mcp-tools-istefox`) used to build the `data.json`
path stays a literal inside `connectorShim.js` (it is a fixed property of
this plugin, not a per-invocation input, unlike the vault path). The
existing `OBSIDIAN_PLUGIN_ID` constant in `mcpbGenerator.ts` is deleted; its
only purpose (interpolating into the old template literal) no longer
exists. `mcpbGenerator.test.ts` keeps its existing assertion that the
generated shim's `data.json` path contains
`".obsidian", "plugins", "mcp-tools-istefox", "data.json"` as the
regression guard for this literal.

### 4. Resolve-and-retry moves from startup to per-request

The current shim runs `waitForServer()` once, before ever spawning
`mcp-remote`. The new shim starts reading stdin immediately (no upfront
wait) and only enters the 30 s retry window reactively, the first time a
POST fails with a connection error (server not listening yet, e.g.
Claude Desktop started before Obsidian's window finished opening). This
mirrors the SPEC's edge case "Obsidian starts after Claude: first request
enters the 30 s retry window, replacing the old startup-only
`waitForServer`" and "Obsidian restarts mid-session on a different port:
next request re-reads `data.json` and reconnects transparently". Both are
impossible with a startup-only wait, since a restart happening after the
first successful connection would never trigger the wait again.

Composed per-request worst case: up to 30 s (retry window, connection
never succeeds) + up to 60 s (the retried POST itself times out) ≈ 90 s
before the shim answers with a `-32000` error and stays alive. Flagged
under Risks: the client's own tool-call timeout may be shorter (see
Risks & HITL gates).

### 5. No explicit stdout lock; one batched `write()` call per logical response unit

The Python bridge needs `stdout_lock` because it runs one OS thread per
request, with real preemption between one thread's multi-line write and
another's. Node's event loop is single-threaded: nothing runs between two
statements of the same synchronous call. `runMain()`'s response handler
joins every message it needs to emit for one JSON-RPC message (the matched
response plus any SSE-forwarded notifications) into one newline-delimited
string and issues exactly one `process.stdout.write(...)` call for that
batch. Concurrent in-flight requests can interleave *between* batches
(that's fine, since that's the concurrency the SPEC asks for) but never *within*
one, so no lock primitive is needed to get the same serialization guarantee
Python achieves with an explicit `threading.Lock`.

## Alternatives considered

**A. Keep the shim as a TS template-literal string inside `mcpbGenerator.ts`
(grow today's `buildShim()` in place), rather than a standalone `.js`
file.** Rejected: this is what SPEC explicitly rules out ("New shim source
as a standalone, unit-tested `.js` file"), and for a structural reason
beyond just following instructions: a template-literal string cannot be
`bun test`-imported and exercised as real code. The SSE parser, the
line-framing reducer, and the error-envelope builder would each need either
(a) `eval()` of the template string in tests (fragile, and it's exactly the
kind of dynamic code execution this whole feature exists to get away from
in spirit) or (b) a parallel, hand-maintained set of TS helper functions
kept behaviorally identical to the string template by manual review. This is a
permanent drift risk between "the code we test" and "the code we ship" for
logic that will be debugged and iterated on (SSE edge cases, timeout
tuning) more than the icon ever will be.

**B. Load `connectorShim.js` at build time via a Bun `{ type: "text" }`
import attribute (`import CONNECTOR_SHIM_SOURCE from "./connectorShim.js" with { type: "text" }`),
skipping the codegen step and the checked-in generated module entirely.**
This would be strictly less code (no `connectorShimSource.ts`, no
`gen-shim-source.ts`, no freshness-guard test, a single source of truth by
construction). Rejected for this codebase specifically, not on general
merit: `bun.config.ts`'s `check` script is `tsc --noEmit`, gating every
build (`"build": "bun run check && bun bun.config.ts --prod"`), and this
repo's build pipeline already carries a long, hard-won list of
narrowly-scoped workarounds (`onnxruntime-node`/`onnxruntime-web`
redirects, `import.meta.url`/`__dirname` placeholder macros, a post-build
string-replace step for Bun's `Object(import.meta).url` emission bug); see
`bun.config.ts`. TypeScript's static understanding of Bun's `type: "text"`
import-attribute loader (does `tsc --noEmit` resolve it as `string` without
an ambient module declaration, and does that declaration exist in the
`bun-types` version pinned here) is unverified for this project's exact
toolchain versions, and a failure mode here is a `bun run check` break,
one of the two hard release gates (SPEC success criterion 5). The icon
precedent (checked-in generated string module) is proven working in this
exact build today; reusing it is the lower-risk choice. If `bun-types`'
`type: "text"` support is later verified solid, revisiting this alternative
to delete the codegen step is a reasonable, low-risk follow-up, but not
inside this change.

**C. True streaming reads of the HTTP response body (`response.body.getReader()`)
instead of `await response.text()`/`.json()`.** Rejected: the Python bridge
this shim mirrors does a single buffered `resp.read()`, and SPEC's "no
shim-side size cap; streamed read" edge case is read as "don't impose an
artificial byte cap," not "must incrementally process chunks". `parseSse`
and `JSON.parse` both need the complete body before they can produce a
result (an SSE event is only complete at its blank-line terminator; a JSON
document is only complete at its closing brace). Chunked reading buys
nothing here and adds a second code path (partial-chunk buffering) with no
corresponding test coverage requirement in SPEC. `response.text()` already
consumes the stream incrementally under the hood with no engineered cap;
that satisfies the stated intent.

**D. Auto-detect and fall back to `npx mcp-remote` if it happens to be on
`PATH`, otherwise use the pure-Node path.** Rejected: reintroduces the
exact two failure surfaces this ADR removes (PATH-dependent behavior that
differs across machines and Claude Desktop versions, and a third-party
dependency in the connection path) for zero benefit. The pure-Node path
handles every case `mcp-remote` did (including SSE, which `mcp-remote`
itself does not parse the way this server needs; that's the whole reason
ADR-0012 exists for the Windows bridge). A conditional fallback also
doubles the states to test and document.

## Consequences

### Positive

- The `.mcpb` connector works under Claude Desktop's actual restricted
  launch environment. The reproduced crash (`spawn npx ENOENT` under
  `PATH=/usr/bin:/bin:/usr/sbin:/sbin`) is structurally impossible: the
  shim spawns no child process at all.
- No third-party code (`mcp-remote`) executes in the `.mcpb`'s connection
  path; only Node built-ins and this project's own reviewed, tested source.
- Survives Obsidian restarts, port fallback, and token rotation mid-session
  without a Claude Desktop restart (per-request resolution replaces
  once-at-startup).
- Failures are diagnosable and non-fatal: JSON-RPC errors surface in
  Claude's UI with a cause instead of an opaque "Server disconnected";
  `mcp-server-*.log` shows lifecycle and error lines.
- `connectorShim.js`'s SSE-parsing core is closely parallel (function names,
  signatures, edge cases) to `scripts/obsidian_mcp_bridge.py`'s pipeline,
  so a future SSE-format change is one mental model applied to two files, not
  two independently-derived implementations to keep in sync by accident.

### Negative

- Two build artifacts must stay in sync (`connectorShim.js`,
  `assets/connectorShimSource.ts`); mitigated by the freshness-guard test
  but that test only fires when `bun test` runs. An editor-only change to
  `connectorShim.js` with no test run before commit ships stale.
- Composed worst-case per-request latency (~90 s: 30 s retry + 60 s
  timeout) before the shim reports failure is long enough that some MCP
  clients' own tool-call timeout could fire first, surfacing a generic
  client-side error instead of the shim's descriptive one in that specific
  slow-reconnect case. Bounded by SPEC's explicit constants; not a defect
  introduced by this design, but worth watching post-release (see Risks).
- The README's Claude Desktop section and the Settings UI's "Download
  .mcpb" button (gated on `nodeStatus.found`, copy reading "Node.js must be
  on your PATH") both describe a Node/PATH requirement that no longer
  applies to the `.mcpb` flow after this change. This is out of this ADR's scope
  (SPEC does not list UI changes), tracked as a fast-follow risk (see
  Risks & HITL gates) rather than folded into this plan, since fixing it
  correctly requires distinguishing the `.mcpb` path (no longer needs
  system Node) from the manual-JSON-config path (`npx mcp-remote`, still
  does).

### Neutral

- `mcpbGenerator.ts`'s public surface (`generateMcpb(input)`,
  `McpbGeneratorInput`) is unchanged. `ClientConfigSection.svelte`'s call
  site needs no edit.
- The `.mcpb` manifest (`manifest.json`) byte shape is unchanged; only
  `server/index.js`'s contents differ. `@anthropic-ai/mcpb validate` in CI
  (`.github/workflows/release.yml`) continues to validate the same
  manifest schema.
- No change to the Obsidian-side HTTP server, the Windows Python bridge, or
  the manual-JSON-config code paths (`generators.ts`, `claudeDesktop.ts`),
  all explicitly out of SPEC's scope and untouched by this ADR.

## Addendum (0.27.2): MCP-Protocol-Version echo, progress notifications, outputSchema

A spec compliance review against modelcontextprotocol.io surfaced three gaps and one
forward-looking hardening opportunity, all closed in this release.

**`MCP-Protocol-Version` header, shim to server.** The spec requires clients to echo the
negotiated protocol version on every request after `initialize`. The shim now extracts
`result.protocolVersion` from the `initialize` response (falling back to
`PROTOCOL_VERSION_FALLBACK = "2025-06-18"` if the server omits it) and sends it as
`MCP-Protocol-Version` on every subsequent POST. This mirrors
`scripts/obsidian_mcp_bridge.py`'s existing `set_protocol_version`/`get_protocol_version`
pattern, which already did this correctly for the Windows bridge.

**`MCP-Protocol-Version` header, server-side validation.** Previously this only worked
because `@modelcontextprotocol/sdk`'s `StreamableHTTPServerTransport` validates the
header internally. `mcp-transport/services/middleware.ts` now has a first-party
`checkProtocolVersion` step (absent header allowed, unsupported value rejected with 400),
checked after Origin and before bearer auth, and covered by this project's own tests
instead of relying entirely on SDK internals.

**Progress notifications during the shim's retry window.** The spec allows a client to
reset its own request timeout on receipt of `notifications/progress`. When an incoming
request carries `params._meta.progressToken`, the shim now emits a
`notifications/progress` message on each transport-resolve retry attempt. This is
opt-in and inert for clients that don't request it (no behavior change for the common
case, Claude Desktop's `initialize` request does not carry a `progressToken` today, so
this does not affect the 0.27.1 timeout fix; it hardens the shim for any future request a
client asks to track this way).

**`outputSchema` on tools.** `ToolRegistryClass` gained `setOutputSchemas()`, mirroring
the existing `setAnnotations()` side-channel pattern exactly, surfaced in `tools/list`.
Only `get_vault_file`'s `format=json` response is modeled (the one tool with a genuinely
reusable named result type, `VaultFileJson`); the other ~50 tools each have their own
distinct inline shape and are out of scope for this pass, since no client this project
supports currently uses `outputSchema` for programmatic/code-mode tool calling.

## Addendum (0.27.1): timeout retune

The 30 s retry window / 60 s per-request timeout from the original decision above
collided with Claude Desktop's own MCP client, which uses the standard SDK default
request timeout of 60000ms for every request including `initialize`. Observed in
production: a fresh 0.27.0 install hit this exactly. Claude Desktop sent `initialize` at
17:43:30.071 while Obsidian's vault window was still cold-starting; the shim's single
POST attempt hung for the full `requestTimeoutMs`, and at 17:44:30.075, 60.004 s later,
Claude's own client gave up first (`notifications/cancelled`, `Server transport closed
(renderer released port)`) and disconnected before the shim could answer with its own
descriptive error. Since the two timeouts were equal, the client always won this race on
any cold-start or slow-response scenario.

Fix: `RETRY_WINDOW_MS` 30000 → 20000, and the per-request timeout (now a named constant,
`DEFAULT_REQUEST_TIMEOUT_MS`) 60000 → 25000. Worst composed path (connection error on the
first POST, re-resolve transport, retry once) is now `RETRY_WINDOW_MS +
DEFAULT_REQUEST_TIMEOUT_MS` = 45 s, a 15 s margin under Claude's 60 s ceiling. Each
individual path (transport-resolve-only: 20 s; single-POST-hang: 25 s) is comfortably
under 60 s on its own. A unit test (`connectorShim.test.ts`) pins the sum invariant
directly, so retuning one constant without the other fails the suite.
